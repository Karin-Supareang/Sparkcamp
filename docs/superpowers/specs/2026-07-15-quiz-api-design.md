# Quiz API Design

## Purpose
Expose the existing quiz lifecycle (lecture creation, quiz sessions, question storage,
student responses, AI analytics) over HTTP so a frontend can drive it. Currently
`backend/main.py` is empty and `backend/database/repository.py` has the DB logic but no
HTTP layer.

## Architecture
```
backend/
  main.py                  # FastAPI() app, CORS, include_router(...) x5
  schemas.py                # Pydantic request/response models
  routers/
    __init__.py
    lectures.py             # POST /lectures, GET /lectures/{id}
    quizzes.py               # POST /quizzes, GET /quizzes/{id}, GET /quizzes/by-code/{code}, PATCH /quizzes/{id}/close
    questions.py             # POST /quizzes/{id}/questions, GET /quizzes/{id}/questions
    responses.py             # POST /responses
    analytics.py             # POST /quizzes/{id}/analyze, GET /quizzes/{id}/analysis
```
Each router uses the existing `get_db` dependency from `config/database.py`. CORS allows
all origins for now (no frontend URL exists yet in this repo) — flagged to tighten later.

## Identity / auth
No auth system exists (no User table). `teacher_id` / `student_id` are trusted as plain
strings sent by the client, matching the existing schema. Revisit if/when auth is added.

## Question creation
`POST /quizzes/{id}/questions` accepts a ready-made list of questions (text, options,
correct answer, topic). It does not call an LLM itself — question generation is a
separate concern (not yet built in this codebase).

## Analytics trigger
`POST /quizzes/{id}/analyze` is synchronous: it blocks until the AI call
(JamAI/Qwen3, via `AnalyticsService`) finishes and returns the report directly, same
pattern as `test_ai_workflow.py`. Async/polling can be added later if runtime becomes a
problem.

## Endpoints

| Method & path | Purpose | Repo function |
|---|---|---|
| `POST /lectures` | Teacher uploads slide -> creates lecture | `create_lecture` (existing) |
| `GET /lectures/{lecture_id}` | Fetch one lecture | `get_lecture_by_id` (new) |
| `POST /quizzes` | Start a quiz session for a lecture | `create_quiz_session` (existing) |
| `GET /quizzes/{quiz_id}` | Quiz status (open/closed, lecture link) | `get_quiz_by_id` (new) |
| `GET /quizzes/by-code/{quiz_code}` | Student joins via QR/code | `get_quiz_by_code` (new) |
| `PATCH /quizzes/{quiz_id}/close` | Teacher ends the quiz | `close_quiz_session` (existing) |
| `POST /quizzes/{quiz_id}/questions` | Save a pre-made question set | `insert_generated_questions` (existing) |
| `GET /quizzes/{quiz_id}/questions` | List questions for a quiz, correct_answer stripped | `get_questions_by_quiz` (new) |
| `POST /responses` | Student submits an answer, returns is_correct | `submit_student_response` (existing) |
| `POST /quizzes/{quiz_id}/analyze` | Run the AI analytics workflow (sync) | wraps `AnalyticsService` |
| `GET /quizzes/{quiz_id}/analysis` | Fetch a previously saved report | `get_analysis_result` (new) |

## Error handling
- 404 when lecture/quiz/question/analysis not found.
- 400 for invalid input (Pydantic validation handles most of this automatically).
- 409 if `quiz_code` already exists (unique constraint violation caught and reported).
- 400 if submitting a response to a quiz that is already closed.
- 502 if the AI analytics call fails, with the underlying error message.

## Testing
No frontend exists yet, so verification is via FastAPI's autogenerated `/docs` (Swagger
UI) — manually exercise the golden path (create lecture -> create quiz -> add questions
-> submit a response -> close quiz -> analyze -> fetch analysis).
